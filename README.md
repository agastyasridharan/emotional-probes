# Emotional Probes

Extract, validate, and steer **emotion** and **emotion-deflection** probes in large language models — a faithful, open-model **methodology port** of Anthropic's ["Emotion Concepts and their Function in a Large Language Model"](https://transformer-circuits.pub/2026/emotions/index.html) (Sofroniew, Kauvar, Saunders, Chen, et al., April 2026).

![Emotion Visualiser Demo](assets/demo.gif)

> The paper studies **Claude Sonnet 4.5** (closed weights). This repo runs the *same methodology* on an **open** model you choose (default `Qwen/Qwen2.5-7B-Instruct`), so numbers won't match the paper exactly — but every technique does. The conceptual companion to the code is **[`docs/PAPER_DEEP_READ.md`](docs/PAPER_DEEP_READ.md)**, a full multidisciplinary read-through; **[`docs/ARCHITECTURE.md`](docs/ARCHITECTURE.md)** maps every paper experiment to its module.

## What this gives you

1. **Generate** synthetic datasets for 171 emotion concepts — emotional stories, neutral baselines, and emotion-deflection dialogues (via an external API **or** self-generated by the probed model itself).
2. **Extract** residual-stream activations through any HuggingFace causal LM.
3. **Compute** emotion probes (difference-of-means directions with PCA confound removal) and deflection probes (additionally orthogonalised against the expression space).
4. **Validate & characterise** the probes — logit lens, implicit scenarios, intensity tracking, geometry (valence/arousal), layerwise division of labour, preferences (Elo + steering), and more.
5. **Apply** them to alignment-relevant behaviours — reward hacking, sycophancy/harshness, blackmail, post-training drift.
6. **Visualise** per-token probe activations on arbitrary text in a browser.

## Design in one paragraph

All the maths is **pure NumPy** and lives in `emotion_probes/core/`, so it is unit-tested and runs on a laptop with no GPU. Everything that needs a GPU (loading the model, forward passes, generation, steering) is isolated behind one class, `ProbedModel`, which imports torch **lazily**. The whole package therefore imports and tests on CPU; you only need the cluster to actually run the experiments. A single frozen [`Config`](emotion_probes/config.py) is the source of truth for the model id, the analysis layer (stored as a **fraction of depth**, default 2/3, so it's correct for any model), paths, and every hyper-parameter.

**Swapping models:** set `Config.model_id` to any mainstream HuggingFace decoder LM with a fast tokenizer (Llama, Qwen, Mistral/Mixtral, Gemma, Phi, GPT-NeoX/Pythia, OPT, Falcon, MPT/DBRX, …). Vectors are stamped with the model that built them and a guard refuses to read a bank against a different model, so **rebuild the vectors when you change the model**. The precise support matrix and caveats (slow tokenizers, base vs chat models, state-space models) are in [`docs/ARCHITECTURE.md` → Model compatibility](docs/ARCHITECTURE.md#model-compatibility--can-i-pass-an-arbitrary-open-model). For *which specific* open models to run, in what order, and why — with the 4×H100 capacity math and per-model extraction settings — see [`docs/MODELS.md`](docs/MODELS.md).


## Quickstart (the pipeline, end to end)

Paths come from `Config`; set `EMOTION_PROBES_DATA=/data/<you>/emotion-probes` so big artifacts don't land in `$HOME`. Full cluster instructions are in **[`docs/DEPLOY.md`](docs/DEPLOY.md)**.

```bash
# 1. Generate datasets (external API; set EMOTION_PROBES_SELF_GENERATE=1 to use the probed model)
python -m emotion_probes.generation.generators emotion_stories     # emotional stories
python -m emotion_probes.generation.generators neutral_stories     # neutral story baseline
python -m emotion_probes.generation.generators neutral_dialogues   # neutral dialogue baseline

# 2. Extract activations (GPU), then build the EXPRESSION vectors
python -m emotion_probes.extraction.extractor stories
python -m emotion_probes.extraction.extractor neutral_stories
python -m emotion_probes.pipeline emotion                          # expression vectors

# 3. Deflection needs the expression vectors first (to pick dissimilar emotion pairs)
python -m emotion_probes.generation.pair_selection                 # choose masking pairs
python -m emotion_probes.generation.generators deflection          # deflection dialogues
python -m emotion_probes.extraction.extractor neutral_dialogues
python -m emotion_probes.extraction.extractor deflection
python -m emotion_probes.pipeline deflection                       # deflection vectors

# 4. Visualise (forward a port from the GPU box, then open it locally)
python visualise.py
```

## Running the experiments

Each experiment is a small class with a `run()` method that returns a results `dict` and writes JSON + plots under `config.analysis_dir`. They load the model and the probe bank, so they need a GPU. Examples:

```bash
python -m emotion_probes.analysis.logit_lens          # Table 1: tokens a vector promotes
python -m emotion_probes.analysis.geometry            # Figs 5-9: clusters, PCA, RSA
python -m emotion_probes.analysis.preferences         # Fig 4: Elo preferences + steering
python -m emotion_probes.alignment.reward_hacking     # Figs 30-31: desperation -> hacking
python -m emotion_probes.alignment.sycophancy         # Figs 32-35: sycophancy/harshness
```

Or from Python:

```python
from emotion_probes import Config
from emotion_probes.models import ProbedModel
from emotion_probes.core.probes import ProbeBank
from emotion_probes.analysis import GeometryAnalysis

config = Config()                                  # or Config().with_(model_id="meta-llama/Llama-3.1-8B")
bank = ProbeBank.load(config.emotion_vectors_path) # geometry needs no GPU
print(GeometryAnalysis(bank, config).run())
```

## How the probes work

**Expression probes.** For each of 171 emotions we generate stories where a character *experiences* the emotion **without ever naming it** (the construct-validity safeguard). We average residual-stream activations per emotion (from the 50th token), subtract the global mean, project out confound directions learned from neutral text (PCA, 50% variance), and unit-normalise. The result detects when an emotion is openly expressed.

**Deflection probes.** We generate dialogues where a speaker *masks* one emotion with another, extract activations on the masking speaker's tokens (using **exact** character-offset → token alignment), apply the same difference-of-means recipe, then additionally orthogonalise against the expression-vector space (99% variance). The result detects an emotion that is contextually *present but suppressed* — a distinct, alignment-relevant signal.

## What changed from the original repository

This is a ground-up refactor onto a clear, class/function-based package, fixing eight isolated issues while **preserving all functionality** (including the full visualiser UI). The headline fixes:

| # | Issue | Fix |
|---|-------|-----|
| 1 | Model hard-coded (Gemma) in ~8 files | One swappable `Config.model_id`; `ProbedModel` auto-detects layers across architectures |
| 2 | Stories written by a *different* model than the one probed | `self_generate` backend lets the probed model author its own stories (the paper's design) |
| 3 | No in-repo validation of the vectors | A whole `analysis/` + `alignment/` suite (logit lens, OOD, preferences, steering, classification…) |
| 4 | Analysis layer arbitrary (23 here, 19 there) | `layer_fraction` (default 2/3) → correct index for any depth |
| 5 | `select_pairs.py` doc/code mismatch | Single documented `dissimilar_fraction = 0.25` |
| 6 | 12 stories per single API call (homogeneity) | `one_story_per_call` mode |
| 7 | Approximate token-span attribution | Exact `offset_mapping`-based spans |
| 8 | Correct hygiene niceties | Preserved (neutral-dialogue PCA; Human/Assistant relabel) |
